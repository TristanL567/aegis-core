# Provider Operations Context

- Provider knowledge is reference material here, not a cloud skill, provider
  runtime doc, or activation path.
- AWS, Azure, GCP, Kubernetes, and serverless platforms use different names for
  similar concepts such as accounts, projects, subscriptions, roles, service
  accounts, managed identities, policies, and resource groups.
- Managed services hide some runtime layers while exposing provider-specific
  limits, logs, events, quotas, identity models, and failure modes.
- Kubernetes context may include deployments, pods, replica sets, services,
  ingress, config maps, secrets, service accounts, namespaces, probes, and
  events.
- Serverless context may include cold starts, concurrency, timeouts, event
  sources, retries, dead-letter queues, environment variables, and identity
  bindings.
- Provider-specific interpretation should stay tied to evidence from the
  affected platform rather than becoming a generic runbook.
